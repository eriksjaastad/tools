  
**Multi-Layer Agent Delegation**

**Architecture**

A Model-Agnostic Delegation Protocol for Hierarchical Agent Trees

**Synth Insight Labs**

Architecture Notes — March 2026

Erik Sjaastad, Conductor

# **1\. The Core Problem**

You have agents running on different runtimes (Claude Code, OpenClaw, Gemini CLI, Kiro, Augment) across different machines (Mac mini, laptop, potentially cloud). Each runtime has its own way to spawn a sub-agent and get results back. There is no shared protocol.

The question is: can you define a universal delegation contract that works regardless of what model or runtime sits behind each role in the tree?

**Short answer:** Yes, but not by standardizing the spawn mechanism. Instead, you standardize the envelope — what goes in and what comes out. The spawn mechanism stays runtime-specific.

# **2\. How the Tree Works**

The hierarchy follows a four-tier model that maps directly to organizational structure:

| Layer | Role | Responsibility | Analogy |
| :---- | :---- | :---- | :---- |
| 0 | Conductor (Erik) | Sets goals. Never executes. | CEO |
| 1 | Architect | Decomposes goals into tasks. Assigns to Floor Managers. | VP / Director |
| 2 | Floor Manager | Manages a bounded scope. Spawns and supervises workers. | Team Lead |
| 3 | Worker | Executes a single, well-scoped task. Returns a result. | IC Engineer |

### **Key Structural Rules**

* **Workers never spawn sub-agents.** They execute and return. If a task is too big, the Floor Manager should have broken it down further.

* **Floor Managers own a bounded scope.** A FM managing “code quality” should not be touching deployment. Scope boundaries prevent delegation loops.

* **The Architect is the only layer that sees the full goal.** Floor Managers receive tasks, not goals. This is the critical distinction.

* **The Conductor (you) never writes code.** You direct via Slack, Claude CLI, or whatever orchestration interface sits at the top.

# **3\. The Delegation Contract**

This is the core of the architecture. Every delegation, at every layer, follows the same shape regardless of runtime.

## **3a. The Task Envelope (Down)**

When a higher layer delegates to a lower layer, it sends a Task Envelope:

| Field | Type | Description |
| :---- | :---- | :---- |
| task\_id | string | Unique identifier. Generated by the delegator. |
| goal | string | What success looks like. Not how to do it. |
| context | string\[\] | Relevant background. File paths, prior decisions, constraints. |
| acceptance\_criteria | string\[\] | How the delegator will judge the result. |
| constraints | object | Budget (tokens/cost), time limit, allowed tools, forbidden actions. |
| output\_schema | string | What shape the result should be in (free text, JSON, file path, etc.). |
| parent\_task\_id | string? | Optional. Links to the parent task for tracing. |

## **3b. The Result Envelope (Up)**

When a lower layer completes work, it returns a Result Envelope:

| Field | Type | Description |
| :---- | :---- | :---- |
| task\_id | string | Matches the incoming task\_id. |
| status | enum | completed | failed | blocked | partial |
| result | any | The deliverable. Matches the requested output\_schema. |
| artifacts | string\[\] | File paths, URLs, or references to created artifacts. |
| cost | object | Tokens used, API cost, wall time. |
| notes | string? | Anything the delegator should know (blockers, assumptions, warnings). |
| child\_tasks | object\[\]? | If this agent delegated further, a summary of sub-task results. |

**Why this works:** The envelope is completely agnostic about what sits behind it. A Worker running on Claude Code via \-p returns the same shape as a Worker running on Gemini \--yolo or an OpenClaw sessions\_spawn. The runtime adapter translates the envelope into whatever spawn mechanism the runtime uses, and translates the response back.

# **4\. Runtime Adapters: How Each System Spawns**

The delegation contract stays the same. The adapter is the thin layer that translates.

| Runtime | Spawn Mechanism | Input | Output | Notes |
| :---- | :---- | :---- | :---- | :---- |
| Claude Code | claude \-p "..." \--output-format json | Prompt string via stdin/args | JSON on stdout | Use \--session-id for multi-turn. \--max-turns for budget. |
| OpenClaw | sessions\_spawn via Slack/API | Slack message or API call | Message back to channel | Already has native sub-agent system. |
| Gemini CLI | gemini \--yolo "..." | Prompt string | stdout text | Less structured. Needs wrapper for JSON output. |
| Kiro / Augment | IDE-native task delegation | Task pane or prompt | File changes \+ response | Harder to automate headlessly. |
| Python subprocess | subprocess.run(\[...\]) | Any CLI agent | stdout capture | Universal fallback for any CLI agent. |
| HTTP/API | POST to agent endpoint | JSON body (Task Envelope) | JSON response (Result) | For cross-machine or cloud agents. |

**The adapter pattern:** Each runtime gets a thin adapter that accepts a Task Envelope, translates it into the runtime’s native spawn call, captures the output, and packages it back into a Result Envelope. The Floor Manager doesn’t need to know which runtime a Worker uses.

# **5\. What Existing Frameworks Do (and Don’t Do)**

Each major framework solves a piece of this problem. None solve your exact problem because your agents aren’t all running inside one framework.

### **OpenAI Swarm / Agents SDK**

* **Model:** Flat peer handoffs. Agent A calls transfer\_to\_agent\_b() and control passes. Stateless between calls.

* **Delegation:** Handoff functions return an Agent object. The framework swaps the system prompt and tools.

* **What’s useful:** The idea that a handoff is just a function call that returns an agent. Simple and composable.

* **What’s missing:** No hierarchy. No result aggregation. No cross-machine. Swarm is now deprecated in favor of the Agents SDK, which adds sessions and guardrails but is still OpenAI-only.

### **CrewAI (Hierarchical Process)**

* **Model:** Manager agent decomposes goals and delegates to specialist workers. Closest to your Conductor → Architect → FM → Worker model.

* **Delegation:** Manager LLM decides which worker to invoke. Workers have allow\_delegation=False to prevent loops.

* **What’s useful:** The role/goal/backstory pattern for agent identity. The hierarchical process type. The idea of a manager\_llm separate from worker LLMs.

* **What’s missing:** In practice, the hierarchical mode has documented issues — it often runs tasks sequentially rather than truly delegating. Token cost scales fast (each delegation \= 3+ LLM calls). Python-only. Single machine.

### **AutoGen (Microsoft)**

* **Model:** Conversational agents. Agents talk to each other in multi-turn chat to jointly solve tasks. v0.4 introduced an actor-model architecture.

* **Delegation:** GroupChat with a manager that selects the next speaker. Or hierarchical chat where a supervisor routes.

* **What’s useful:** The conversational paradigm — agents negotiate rather than receive orders. The v0.4 event-driven, async architecture is production-grade. Model-agnostic.

* **What’s missing:** Complexity. The layered architecture (Core → AgentChat → Extensions) is powerful but heavy. No native cross-machine support without custom transport.

### **LangGraph (LangChain)**

* **Model:** Graph-based orchestration. Agents are nodes, edges define control flow with conditional branching, loops, and parallel execution.

* **Delegation:** Supervisor pattern via create\_supervisor(). Handoff tools route to sub-agents. Orchestrator-worker with Send API for parallelism.

* **What’s useful:** Explicit state management. The supervisor-of-supervisors pattern maps to multi-level hierarchies. Built-in persistence and streaming.

* **What’s missing:** Graph complexity. Debugging distributed async agents is hard. Primarily Python. Heavy dependency chain.

### **Google A2A Protocol**

* **Model:** Open standard for agent-to-agent communication. Client agents send tasks to remote agents via HTTP/JSON-RPC. Agents discover each other via Agent Cards (JSON at /.well-known/agent.json).

* **Delegation:** Task-oriented. Client sends a task, remote agent works on it, sends back artifacts. Supports long-running tasks via SSE streaming. Now at v0.3 with 150+ supporting organizations.

* **What’s useful:** THIS IS THE CLOSEST THING TO YOUR UNIVERSAL CONTRACT. A2A defines exactly what you need: a task envelope, a result with artifacts, agent discovery, and it’s framework-agnostic. Built on HTTP/JSON-RPC, works cross-machine by design.

* **What’s missing:** It’s designed for web-scale enterprise agents, not a van-based Mac mini talking to a laptop over Starlink. Overhead of running HTTP servers for each agent may be too heavy for your setup.

### **Claude’s Native tool\_use**

* **Model:** An LLM call can include tool definitions. The model decides to call a tool, you execute it, feed the result back. Sub-conversations are just nested API calls.

* **Delegation:** Define a “delegate\_task” tool. When the model calls it, your code spawns a sub-agent (via any mechanism), gets the result, and returns it as tool output.

* **What’s useful:** Elegant. The delegation is just another tool call. Works with any model that supports tool\_use. Claude Code already uses this internally for its Task() sub-agents.

* **What’s missing:** You have to build the orchestration yourself. No built-in hierarchy, state management, or error handling.

# **6\. The Minimum Viable Tree**

The simplest end-to-end tree that proves the architecture works:

**Conductor (Erik)** → **1 Floor Manager** → **2 Workers**

## **What Needs to Exist**

| Component | What It Is | Build Priority |
| :---- | :---- | :---- |
| Task Envelope schema | JSON schema defining goal, context, acceptance\_criteria, constraints, output\_schema | 1 — First |
| Result Envelope schema | JSON schema defining status, result, artifacts, cost, notes | 1 — First |
| Claude Code adapter | Shell script or Python that takes a Task Envelope, calls claude \-p, parses JSON output into Result Envelope | 2 — Second |
| Floor Manager prompt | System prompt that knows how to decompose a task, spawn workers via delegate\_task tool, and aggregate results | 3 — Third |
| Worker prompt | System prompt for a focused specialist. Receives task, executes, returns result. No delegation. | 3 — Third |
| Orchestrator script | The outer loop: reads the goal from Conductor, sends Task Envelope to FM, collects Result Envelope | 4 — Last |

## **End-to-End Flow**

1. **Step 1:** Conductor writes a goal in Slack (or a file, or a CLI prompt): “Refactor the auth module to use JWT instead of session cookies.”

2. **Step 2:** Orchestrator script reads the goal, wraps it in a Task Envelope, and sends it to the Floor Manager (running as a claude \-p call with the FM system prompt).

3. **Step 3:** Floor Manager LLM decomposes the goal into two sub-tasks: (a) Update the auth middleware, (b) Update the test suite. It calls delegate\_task twice.

4. **Step 4:** For each delegate\_task call, the adapter spawns a Worker (another claude \-p call with a Worker system prompt and the sub-task envelope).

5. **Step 5:** Each Worker executes, returns a Result Envelope (status, artifacts, cost).

6. **Step 6:** Floor Manager receives both results, synthesizes them, returns a combined Result Envelope to the orchestrator.

7. **Step 7:** Orchestrator reports back to Conductor: “Goal complete. 2 sub-tasks executed. Here’s the summary.”

# **7\. Cross-Machine Delegation**

Can a Floor Manager on the Mac mini spawn a Worker on the laptop, or vice versa?

**Yes**, but the transport layer matters. Here are the options ranked by complexity:

| Transport | How It Works | Pros | Cons | Verdict |
| :---- | :---- | :---- | :---- | :---- |
| SSH | ssh laptop "claude \-p '...'" | Simple. Already works. No new infrastructure. | Latency per call. SSH session overhead. Fragile over Starlink. | Good for MVP. Use this first. |
| HTTP API | Flask/FastAPI endpoint on each machine that accepts Task Envelopes | Clean separation. Async-friendly. Maps to A2A concepts. | Need to run web servers. Security. Port management. | Good for v2. Best long-term option. |
| Message Queue | Redis, NATS, or even file-based queue on shared drive | Decoupled. Handles offline/async. Retry built in. | Another piece of infrastructure. Overkill for 2 machines. | Skip unless you scale to 3+ machines. |
| Slack (via OpenClaw) | Post a task to a Slack channel, agent on the other machine picks it up | Already exists\! OpenClaw does this today. | Slack rate limits. Not designed for structured I/O. | Surprisingly viable. Already your native protocol. |

### **Recommended Path**

* **MVP:** SSH. The Mac mini SSHes into the laptop (or vice versa) and runs claude \-p with the Task Envelope as stdin. Structured JSON in, structured JSON out.

* **v2:** HTTP endpoint. Each machine runs a lightweight Flask/FastAPI server that accepts POST /task with a Task Envelope body and returns a Result Envelope. This is essentially a homebrew A2A implementation.

* **v3 (maybe never):** Full A2A protocol. Agent Cards, discovery, SSE streaming. Only if you’re exposing agents to external collaborators.

# **8\. What We Build First**

Ordered by dependency. Each step unlocks the next.

1. **Step 1: Define the schemas.** Write task\_envelope.schema.json and result\_envelope.schema.json. These are the constitution. Everything else is implementation detail.

2. **Step 2: Build the Claude Code adapter.** A Python script (or shell script) that reads a Task Envelope from stdin, constructs a claude \-p call with the right system prompt and constraints, captures the JSON output, and writes a Result Envelope to stdout. Test it with a single Worker.

3. **Step 3: Build a Floor Manager prompt.** A system prompt that receives a Task Envelope, decomposes it, and calls a delegate\_task tool for each sub-task. The delegate\_task tool calls the Claude Code adapter. Test with 1 FM \+ 2 Workers on the same machine.

4. **Step 4: Build the orchestrator.** A script that reads a goal from the Conductor (you), wraps it in a Task Envelope, sends it to the FM, and reports the Result Envelope back. This is the outer loop.

5. **Step 5: Add SSH transport.** Extend the adapter to optionally run claude \-p on a remote machine via SSH. The FM doesn’t know or care — it just calls delegate\_task and gets results.

6. **Step 6: Integrate with OpenClaw.** Build an OpenClaw adapter that translates Task Envelopes into Slack messages and Result Envelopes from Slack responses. Now your existing agents can participate in the tree.

# **9\. Key Design Decisions**

### **Goals vs. Tasks**

The Conductor gives the Architect a goal (“make the auth system use JWT”). The Architect gives Floor Managers tasks (“update auth middleware to issue JWTs”). Floor Managers give Workers atomic tasks (“rewrite the verifyToken function to decode a JWT”). Each layer adds specificity. A Worker should never need to ask “what should I do?”

### **State Management**

Workers are stateless. They receive a Task Envelope, execute, return a Result Envelope. Floor Managers maintain state for the duration of a goal (which sub-tasks are done, which are pending, what results came back). The Architect maintains state across goals. This maps cleanly to your existing patterns: HEARTBEAT.md for state, SOUL.md for identity, AGENTS.md for the roster.

### **Error Handling**

The Result Envelope has a **status** field: completed, failed, blocked, or partial. A Floor Manager that gets a “failed” result should decide: retry? escalate? skip? This is where the scar-tissue SLA applies — every failure mode becomes a checklist item.

### **Cost Control**

The Task Envelope includes a constraints.budget field. A Floor Manager that has burned 80% of its token budget should stop spawning workers and return a partial result. This is your tiered routing in action — use cheap models (Qwen/Ollama) for review tasks, expensive models (Claude) for architectural decisions.

# **10\. The Big Insight**

*Every framework reviewed — Swarm, CrewAI, AutoGen, LangGraph — solves delegation* **within** *a single runtime. Your problem is delegation* **across** runtimes and machines. The frameworks are useful for inspiration (CrewAI’s hierarchical process, LangGraph’s supervisor pattern, AutoGen’s conversational negotiation), but none of them are your solution.

Your solution is simpler than any of them: **a JSON envelope contract \+ thin runtime adapters.** The intelligence is in the prompts (what the FM knows how to decompose, what the Worker knows how to execute), not in the framework. This is “intelligence in the checklist, not the prompt” applied to architecture.

Google’s A2A protocol validates this approach — it’s literally a standardized version of the same pattern: task envelopes, result artifacts, agent discovery, framework-agnostic. The difference is A2A targets enterprise-scale HTTP services, and you need something that works over SSH between a van and a Mac mini.

# **11\. The Project Intake Gate**

The delegation architecture above solves how work flows through the tree. This section solves what work enters the tree in the first place. Without this gate, the tree will efficiently execute the wrong things.

The pattern is a client intake process turned inward. Before any project receives a Task Envelope — before a single token gets spent — it must survive five questions. These questions are designed to catch the specific failure mode where a solo builder burns weeks on infrastructure that nobody asked for, discovers interesting-but-adjacent problems, and never ships the thing that was supposed to make money.

## **The Five Questions**

### **Q1: What does this product DO for the person paying for it?**

*One sentence. No technical words.*

This question forces the project to justify its existence in terms a customer would recognize. If the answer contains the words “pipeline,” “infrastructure,” “refactor,” or “architecture,” the project is serving the builder, not the buyer. That’s not automatically wrong — but it means this is internal tooling, not a product, and it should be prioritized accordingly.

**Good answer:** “It writes your invoice follow-up emails so you don’t have to.”

**Bad answer:** “It provides a multi-agent orchestration layer with tiered LLM routing and a shared memory database.”

### **Q2: What’s the shortest path to someone giving us money?**

*Not the best path. The shortest.*

This question distinguishes between building the product and building the business. The shortest path to revenue is almost never “finish the platform.” It might be “manually do the thing for one paying customer while the automation catches up.” It might be “ship the ugly version that works.” The point is to identify the minimum viable transaction — the smallest unit of value exchange where someone pays for something you made.

If the project has no answer to this question, it is a hobby. Hobbies are fine. But hobbies don’t get the Architect’s tokens.

### **Q3: What are we building that the customer never sees?**

*This is where over-engineering hides.*

Make a list. Everything on it needs a justification that traces back to Q1. A shared memory database is invisible to the customer. A D3 visualization of agent state is invisible to the customer. A four-tier token-routing system is invisible to the customer. Each of these might be necessary — but “necessary for what?” is the question. If the answer is “necessary so the builder can manage 30 projects,” that’s an ops tool, and it competes for time against the thing that actually makes money.

The danger signal: when the invisible list is longer than the visible list. That means the project has become its own infrastructure. The builder is building tools to build tools.

### **Q4: If we had to ship in 48 hours, what would we cut?**

*Everything you’d cut is probably not essential.*

This is the most revealing question. The 48-hour constraint is a lie detector. It forces an honest accounting of what’s load-bearing and what’s nice-to-have that metastasized into months of work. If the answer is “we’d cut the tiered LLM routing and just use one model,” that tells you the tiered routing was a research project, not a ship requirement.

The cut list becomes a “Phase 2” backlog. Not deleted — deferred. The critical move is recognizing that deferred is not abandoned. It’s sequenced. The thing that makes money ships first. The thing that makes it elegant ships second.

### **Q5: Are we solving the problem we started with, or a problem we discovered along the way?**

*The Google Sheets auth question.*

This is the scope-creep detector. Every project generates sub-problems: auth flows, database schemas, deployment pipelines, edge cases. Each sub-problem feels urgent because it’s blocking the current task. But “blocking the current task” is not the same as “blocking the product.” Sometimes the sub-problem is a rabbit hole disguised as a prerequisite.

The test: can you state the original problem in one sentence? Does the current work trace back to it in three hops or fewer? If the chain is “automate invoice follow-ups → need Gmail integration → need OAuth flow → need Google Cloud project → need domain verification → need DNS configuration,” you’re five hops deep and the DNS configuration has nothing to do with invoices. That’s the moment to ask: is there a shorter path that sidesteps this entire chain?

## **How This Integrates with the Delegation Architecture**

The five questions are a gate that sits between the Conductor and the Architect. They are Layer 0.5 — the admission control for the tree.

In practice, this means the Task Envelope from Section 3 gets a new optional field at the top level: **intake\_answers**. When the Conductor creates a top-level goal, the intake answers travel with it. The Architect can reference them when decomposing tasks — and critically, can push back if the decomposition starts drifting from the answers.

## **The Meta-Question: Does This Document Pass Its Own Test?**

Fair question. Let’s run it:

**Q1:** This architecture lets one person direct a team of AI agents to finish projects instead of starting them. (The customer is you.)

**Q2:** The shortest path is Smart Invoice Workflow going live. This architecture serves that goal only if it helps SIW ship faster. If it doesn’t, it’s a distraction.

**Q3:** All of it. The entire delegation tree is invisible infrastructure. It only earns its keep if it measurably increases the rate at which visible products ship.

**Q4:** We’d cut the cross-machine transport, the A2A comparison, and the formal schemas. We’d keep the envelope concept and one Claude Code adapter. That’s the MVP.

**Q5:** The original problem was “how do I get my agents to coordinate.” This document answers that. But the real problem is “how do I ship the things that make money.” The intake gate is the bridge between the two.
